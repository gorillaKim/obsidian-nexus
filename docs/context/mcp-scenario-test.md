# MCP 시나리오 테스트 결과

> 테스트일: 2026-03-19
> 시나리오: 에이전트가 개발 중 필요한 문서를 MCP로 탐색

## 테스트 시나리오

"Phase 3 벡터 검색 개발 중, 임베딩 관련 설계 문서가 필요한 상황"을 가정하고 MCP 도구만으로 문서를 탐색한 후, 동일 작업을 파일시스템 도구(Glob/Grep/Read)로 수행하여 비교.

## 사용한 MCP 도구 및 탐색 흐름

| Step | 도구 | 목적 | 결과 |
|------|------|------|------|
| 1 | `nexus_list_projects` | 볼트 목록 파악 | 3개 프로젝트 확인 |
| 2 | `nexus_search` x3 (병렬) | 키워드로 관련 문서 탐색 | 임베딩: 5건, ONNX: 2건, 아키텍처: 5건 |
| 3 | `nexus_get_section` x2 (병렬) | 필요한 섹션만 정밀 조회 | 임베딩 설정 + 아키텍처 확인 |
| 4 | `nexus_get_backlinks` + `nexus_get_links` (병렬) | 그래프 탐색으로 관련 문서 발견 | 백링크 3건, 포워드링크 2건 |
| 5 | `nexus_get_section` + `nexus_get_metadata` (병렬) | 백링크 발견 문서 심화 조회 | 벡터 검색 설계 상세 확인 |
| 6 | `nexus_resolve_alias` | alias로 문서 직접 접근 | "Search System" → 02-검색-시스템.md |

## 검색 품질 비교 (Before → After 개선)

개선 전(v1)과 개선 후(v2) MCP 검색 결과 비교:

| 검색어 | v1 (개선 전) | v2 (개선 후) | Grep (파일시스템) |
|--------|-------------|-------------|-----------------|
| "embedding 임베딩 벡터" | 2건 | 5건 (+3) | 28줄 (중복 다수) |
| "ONNX model 모델" | 0건 | 2건 (+2) | 0건 |
| "아키텍처 설계 architecture" | 0건 | 5건 (+5) | 파일명으로 발견 가능 |
| "아키텍처" (단독) | 미테스트 | 5건 | 파일명으로 발견 |

### 개선 내역

1. **파일명/제목 FTS5 주입** — 인덱싱 시 첫 번째 청크에 문서 제목+파일명 주입
2. **복합 쿼리 토큰 분리** — 공백 구분 개별 토큰 OR 매칭 추가
3. **MCP tool description 정확화** — 하이브리드/벡터 모드 명시
4. **enrich 배치 쿼리 스코핑** — 전체 테이블 스캔 → doc_ids IN 조건
5. **snippet 길이 확장** — 32 → 64 토큰

## 토큰 사용량 비교

| 항목 | MCP | 파일시스템 (Grep+Read) |
|------|-----|----------------------|
| 검색 3건 병렬 | ~600 tokens | ~1,500 tokens |
| 문서 상세 조회 | ~500 tokens (섹션만) | ~3,800 tokens (파일 전체) |
| 관련 문서 탐색 | ~300 tokens (backlinks) | 불가능 or grep ~500 tokens |
| **합계** | **~1,400 tokens** | **~5,800 tokens** |
| **절약률** | **76%** | baseline |

## MCP 고유 가치 (파일시스템 대체 불가)

- **섹션 단위 조회** (`nexus_get_section`): 파일 전체 읽기 대비 토큰 대폭 절약
- **그래프 탐색** (`nexus_get_backlinks`, `nexus_get_links`): Obsidian 위키링크 구조 활용
- **크로스 볼트 검색**: 여러 프로젝트 동시 검색
- **Alias 해소** (`nexus_resolve_alias`): 영문 별칭으로 한글 문서 접근
- **메타데이터 리랭킹**: backlink_count, view_count 기반 인기도 부스트
